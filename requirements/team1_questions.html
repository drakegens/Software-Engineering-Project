Questions

A. Actors

 A.1. End Users

  A.1.1 What human users, if any, does the system need to accommodate?

  A data entry user and a user running the normalization process on a batch of addresses.

  A.1.2 Will the system need to accommodate different classes of users with different permissions levels?

   No, it will not need to manage permissions between the two types of users.

 A.2 Coupled Systems

  A.2.1 What external systems, if any, does the system need to interoperate with?

   It will not need to interoperate with any external systems.

B. Typical Functionality

 B.1 Workflow

  B.1.1 How does the system receive an address to be normalized?

  A CSV file in the batch mode and manual data entry in interactive mode.

  B.1.2 What results need to be returned or displayed after the system has processed and normalized an address?

  An output CSV file of the normalized addresses.

  B.1.3 How should information about address receiving, address processing and results be communicated to human users?

  An output CSV file of the normalized addresses.

   B.1.3.1 If automated email, how should that email be formatted and which users should receive it?

   The system should not use email.

   B.1.3.2 If the program needs a GUI, should be in the form of a web application or native application?

   The system should not use a web interface.

  B.1.4 Should information displayed be dependent on the class of the user, if any exist?

  There are not different classes of users in the system.

   B.1.4.1 If so, what which users need to see what?

   See above answer.

  B.1.5 R1 Should the output contain only normalized addresses?

  Using multiple out files is acceptable.

  B.1.6 R1 Does the system need to be able to handle business addresses?

  Yes the system must be able to process business addresses. Users will specify if an address is personal or business..

 B.2 Address Handling Specifics

  B.2.1 If an aspect of address is found to be invalid and repair is impossible, what should be done with the address?

  The user should be notified and given a chance to correct it.

  B.2.2 If two aspects of an address such as zip code and state are contradictory, what should be done with the address?

   A best attempt should made to normalize it if possible.

  B.2.3 If non-alphanumeric characters are found within address how should they be handled?

  They should be ignored if they would affect processing.

  B.2.4 Is it appropriate to attempt to correct spelling errors in certain fields such as state or street address ending noun?

  Yes, as long as you do not invalidate a potentially valid address.

  B.2.5 R1 If a duplicate normal address is found should both be preserved?

  If a duplicate is found it can be discarded.

C. Use Cases

 C.1 Batch Job

  C.1.1 What kind of scheduling interfaces does the system need in this mode?

  Yes, users should be notified of how an address was changed.

  C.1.2 R1In batch mode should the system prompt for normalization resolution?

  No, users should be notified of how an address was changed.

 C.2 Data Entry and Correction

  C.2.1 Do we actively notify users about the corrections we make?

  Yes, users should be notified of how an address was changed.

  C.2.2 What role should the user play in handling addresses that cannot be normalized?

  The user should be notified of addresses that can not be normalized and given a chance to correct.

D. Non-functional

 D.1 Environment

  D.1.1 What operating system does the system need to run on?

  There is no preference for target operating system.

  D.1.2 Is it safe to assume that incoming data from other systems has been validated to have any null fields marked as such and a consistent field ordering?

  Some fields may be empty such as zip four.

 D.2 Constraints

  D.2.1 Does the system need to handle security measures such as encrypting communication or authenticating users?

  The system does not need to handle any security measures.

  D.2.2 Is there a performance target in terms of normalized address per second or similar that needs to be met by the system?

  The system must be able to process a 50,000 address working set.

  D.2.3 R1 Does the system need to preserve the ordering of addresses?

  Yes, the system needs to preserve the ordering of addresses.

 D.3 Licensing

  D.3.1 If this project is to be open-sourced, which license would be most appropriate given how it will be used?

  There are no restrictions on what open source license could be used.
